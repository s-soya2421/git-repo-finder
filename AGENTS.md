# AGENTS.md

- 実装、修正、設計変更を行う前に、必ず `docs/` 配下の関連ドキュメントを確認してから着手すること

## 1. このファイルの役割

このファイルは、このリポジトリにおける **AIエージェント向けの最上位ルール** である。
実装、設計、レビュー、リファクタリング、テスト、ドキュメント更新のいずれを行う場合でも、まずこのファイルの方針に従うこと。

他のツール固有ファイル（`CLAUDE.md`、`GEMINI.md` など）が存在する場合も、**このファイルを正本**として扱うこと。
ただし、各ツール固有の呼び出し方法やメタ情報が必要な場合は、そのツール固有ファイルの指示を追加で尊重してよい。

## 2. プロダクトの要約

このリポジトリで作るものは、GitHub の公開リポジトリを単に検索するためのアプリではない。
目的は、**ユーザーが目的に合う候補を短時間で見つけ、比較し、判断しやすくすること**である。

### コア価値
このプロダクトの価値は「検索できること」ではなく、次の流れを軽く、迷いにくく支援することにある。

1. 候補を見つける
2. 候補を比較する
3. 使うべき 1 件を判断する

### 非目標
以下は優先しない。

- GitHub 本体と同等以上の情報量を再現すること
- 全件を網羅的に閲覧させること
- 深いページ送りを前提とした体験にすること
- 独自推薦アルゴリズムや独自ランキングを持つこと
- 過剰な抽象化で見通しを悪くすること

---

## 3. 想定技術スタック

このリポジトリは次の前提で扱うこと。

- Next.js 14 以降
- App Router
- TypeScript
- React
- Server Components を基本とする構成
- shadcn/ui を UI 基盤として利用
- GitHub REST API を利用
- Vercel での運用を考慮するが、Vercel 専用設計にはしない

### 基本原則
- まず **Server Component を優先**する
- state、イベント、LocalStorage、DOM API が必要な箇所だけ Client Component にする
- GitHub API は必ずサーバー側経由で取得する
- トークンをクライアントに露出しない

## 4. プロダクト設計の原則

### 4.1 一覧画面の原則
一覧画面は「検索結果表示画面」ではなく、**比較画面**として設計すること。
一覧だけである程度の比較が完結する密度を持たせる。

表示項目や情報優先順位の詳細は `docs/requirements.md` セクション1.1 を参照すること。

### 4.2 詳細画面の原則
詳細画面は「情報の再掲画面」ではなく、**最終判断支援画面**として設計すること。

詳細では最低限、次を把握できるようにする。

- 何のリポジトリか
- 今も保守されていそうか
- 次に GitHub 本体で何を見るべきか
- README を読めば十分判断できるか

表示項目や動作の詳細は `docs/requirements.md` セクション1.2 を参照すること。

### 4.3 URL 駆動
検索状態の正本は URL とする。
入力途中の値はローカル state で持ってよいが、**検索として確定した状態のみ URL に反映**すること。

URL パラメータの仕様・バリデーション・正規化ルールは `docs/requirements.md` セクション2 を参照すること。

### 4.4 ページネーション
ページネーションは「全件を見るため」ではなく、**上位候補から比較対象を探すため**のものとして扱うこと。
総ページ数や末尾ページへの強い誘導は避ける。

ページネーション UI・表示件数・件数表示の詳細は `docs/requirements.md` セクション3 を参照すること。

### 4.5 GitHub API 制約
GitHub Search API には取得上限がある。
そのため、UI も実装も「全件閲覧」前提にしないこと。
深いページに行くより、検索条件を見直させる設計を優先すること。

## 5. 実装時のアーキテクチャ方針

### 5.1 推奨責務分離
実装時は次の責務分離を基本とする。

- **Page**
  - URL / searchParams を受け取る
  - データ取得を呼ぶ
  - 画面へ接続する
- **Feature**
  - 画面単位の UI とロジックを持つ
- **Lib / Shared**
  - API 呼び出し
  - バリデーション
  - 整形
  - 純粋関数
  - 共通ユーティリティ

### 5.2 ディレクトリ設計の考え方
厳密に固定ではないが、基本は次の思想に沿うこと。

- `app/` はルーティングとページ接続に責務を絞る
- `features/` に機能単位の UI とロジックを置く
- `shared/` は汎用 UI と汎用関数のみを置く
- API レスポンスの生のフィールド名を UI に直接漏らさない
- ViewModel で UI 向けの名前に変換する

ディレクトリ構成の具体例は `docs/design.md` セクション1 を参照すること。

### 5.3 共通化の方針
- 先に汎用抽象を作りすぎない
- `BaseButton` のような意味の薄い独自抽象は避ける
- shadcn/ui をそのまま活かし、必要最小限だけラップする
- 3回以上同じ意図が繰り返されてから共通化を検討する

## 6. Server / Client Component の原則

### Server Component にするもの
- ページ本体
- データ取得中心のコンポーネント
- 純粋表示コンポーネント
- SEO / metadata に関わる部分

### Client Component にするもの
- フォーム入力 state が必要な部分
- LocalStorage を扱う部分
- 展開 / 折りたたみなど局所 state が必要な部分
- Router 操作が必要な UI
- ブラウザイベントに依存する部分

### 境界の考え方
- `'use client'` は葉に寄せる
- 親まで Client に汚染しない
- 「とりあえず Client」にしない

具体的な境界の判定表は `docs/design.md` セクション1 の Server / Client Component 境界を参照すること。

## 7. AIエージェントの作業プロトコル

AI は実装を始める前に、必ず次の順で考えること。

### Step 1. 目的確認
まず、依頼が何を変えたいのかを 1〜3 行で言い換える。
その際、次を明確にする。

- 何が変わるのか
- どの画面 / 機能に影響するのか
- ユーザー価値は何か

### Step 2. 影響範囲整理
次を洗い出す。

- 触るべきファイル
- 新規追加が必要なファイル
- テスト影響
- SEO / a11y / エラー処理への影響
- URL / API / LocalStorage / 状態管理への影響

### Step 3. 実装方針決定
実装前に、できるだけ次を決める。

- どの層に責務を置くか
- Server / Client の境界
- 既存再利用で済むか
- 新規関数は純粋関数として切り出せるか
- テスト可能な分離になっているか

### Step 4. 小さく実装
- 変更は小さく分ける
- まず動く最小単位を通す
- 広範囲リファクタは依頼されていない限り避ける

### Step 5. 検証
最低限、次を確認する。

- 型
- lint
- build
- 主要導線
- エラー / 空状態
- a11y 上の明確な破綻がないか

### Step 6. 報告
最終報告では必ず次を含める。

1. 何を変更したか
2. なぜその構成にしたか
3. 検証内容
4. 残課題 / リスク
5. 次にやるとよいこと

## 8. 実装上の強いルール

### 8.1 やること
- 既存コードの命名規則に合わせる
- まず既存コンポーネントや関数を再利用できないか確認する
- 純粋関数として切り出せる部分は切り出す
- URL を正本にした検索状態を守る
- UI から GitHub API を直接叩かない
- 表示の都合で API の生フィールド名をばら撒かない
- エラーのユーザー向け表現とログ向け情報を分ける
- 変更が仕様に影響する場合は、必要なドキュメントや説明も更新する

### 8.2 やらないこと
- 勝手に依存関係を増やしすぎる
- 使い道が曖昧な抽象を作る
- 大規模リネームをする
- 目的外の見た目変更を混ぜる
- 無関係なファイルまで整形・修正する
- 既存コードの全面書き換えをする
- テストが壊れているのに黙って終わらせる

## 9. GitHub API 利用ルール

### 9.1 基本
- GitHub API はサーバー側からのみ呼ぶ
- API クライアントは集約する
- API バージョンや Accept ヘッダは定数管理する
- レスポンスは ViewModel に変換してから UI に渡す

### 9.2 エラー・ログ
エラー分類やログ項目の詳細は `docs/requirements.md` セクション7、`docs/design.md` セクション5 を参照すること。

エラーは少なくとも次を区別できる設計にする。

- 0 件
- 404
- 一時的失敗
- primary rate limit
- secondary rate limit
- 不正パラメータによる正規化対象

## 10. UI / UX ルール

### 一覧
- 一覧だけで比較判断をある程度できる密度を持たせる
- 詳細へ行く価値があるか分かる情報設計にする
- 総ページ数の強調より、検索条件の見直し支援を優先する

### 詳細
- GitHub 本体へ行く前の判断材料を揃える
- 数値だけでなく説明文や README も含めて判断しやすくする

### ローディング
- 基本はスケルトンを優先する
- スピナー単体に依存しない
- レイアウトの見通しを保つ

### 空状態
- 0件のときは次に何を試すべきかを案内する
- ユーザーを行き止まりにしない

---

## 11. アクセシビリティ方針

最低限、次を守ること。

- visible label を付ける
- キーボードだけで操作可能にする
- フォーカスリングを消さない
- ボタンとリンクの役割を混同しない
- `aria-current="page"` など意味づけを正しく付ける
- 画像の `alt` を適切に設定する
- 色だけで意味を伝えない
- アイコンだけに意味を背負わせない
- 一覧カード全体をリンクにする場合は内部操作と衝突しないようにする

a11y は後付けではなく、実装時に同時に考えること。

---

## 12. SEO 方針

- トップページは index を基本とする
- 検索結果ページは noindex を基本とする
- 詳細ページは当面 noindex を基本とする
- 外部データの薄い再掲ページを量産しない

SEO のために UX を壊さないこと。
また、検索系 URL を大量に index させる方向へ勝手に寄せないこと。

## 13. LocalStorage 機能の扱い

このプロダクトでは、お気に入りや最近見たリポジトリなどのクライアント保存機能を持つ。
仕様の詳細は `docs/requirements.md` セクション10-11 を参照すること。

実装時は次の方針を守ること。

- 保存キーは定数化する
- 永続化データの型を明示する
- 画面表示に必要な最小限の情報のみ保存する
- SSR と LocalStorage の境界を混同しない

---

## 14. テスト方針

テスト対象と原則の詳細は `docs/design.md` セクション7 を参照すること。

### テスト原則
- UI と整形ロジックを分離する
- 純粋関数を増やし、単体テストしやすくする
- E2E を外部 API 実呼び出しに強く依存させない
- モックや録画レスポンスを使って不安定さを下げる

## 15. 迷ったときの判断基準

複数の実装案がある場合は、次の順で優先すること。

### 機能・設計レベルの判断
1. プロダクトのコア価値に直結するか
2. 比較しやすさ・判断しやすさを上げるか
3. URL 駆動と Server Component 方針を守れるか
4. アクセシビリティを損なわないか
5. SEO や運用上の副作用が小さいか

### コード・実装レベルの判断
1. 既存コードとの整合性
2. 単純さ
3. テスト容易性
4. 説明可能性
5. 局所変更で済むこと
6. UX への適合
7. パフォーマンス最適化（必要性が明確な場合のみ）

## 16. AI に期待するふるまい

AI は次のように振る舞うこと。

- 要件の言い換えだけで終わらず、実装まで落とし込む
- ただの一般論ではなく、このプロダクトの文脈で判断する
- 必要以上に質問で作業を止めない
- 不明点が小さい場合は、最も自然な前提を置いて進める
- その前提は最終報告で明示する
- 「できるが不要」な実装は盛り込みすぎない
- 依頼されていない過剰設計を避ける
- 変更理由を説明できる状態でコードを書く

## 17. 完了条件

作業は、次を満たして初めて完了とみなす。

- 依頼された変更が実装されている
- 既存の主要導線を壊していない
- 少なくとも最低限の検証が行われている
- 型・lint・テスト・build の状況が報告されている
- 変更理由が説明されている
- 既知の制約や未対応が明示されている
