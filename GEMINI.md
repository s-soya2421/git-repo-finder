# GEMINI.md

## 1. 位置づけ

このファイルは Gemini CLI / Gemini 系ツール向けの補助指示である。
このリポジトリでは **`AGENTS.md` を最上位の正本** とする。

**Gemini は作業前にまず `AGENTS.md` を読んで、その方針に沿って行動すること。**

このファイルには Gemini 固有の注意点と、重要ルールの要約を記載する。
`AGENTS.md` と矛盾する場合は `AGENTS.md` を優先する。

---

## 2. Gemini 固有の指示

### コード生成の注意点
- Gemini はコード生成時に大きな変更を一度に出力しやすい傾向がある。このリポジトリでは **小さく、段階的な変更** を求める
- 既存コードを全面書き換えせず、差分として最小限の変更を行うこと
- 新しいファイルを作る前に、既存のファイルで対応できないか確認すること

### 実装前の確認手順
1. `AGENTS.md` を読む
2. 変更対象に関連する `docs/` 配下のドキュメントを読む
3. 変更対象の既存コードを読む
4. `AGENTS.md` セクション7 の作業プロトコルに沿って進める

### 依存関係の追加
- 新しいパッケージを追加する前に、既存の依存関係で実現できないか確認すること
- shadcn/ui で提供されているコンポーネントを独自実装しないこと

### Server Component の徹底
Gemini は `'use client'` を付ける前に、本当に Client Component が必要か再確認すること。
判断基準は `AGENTS.md` セクション6 に従う。

---

## 3. プロダクト価値の要約

このプロダクトは GitHub の検索機能を再現するものではない。
目的は、**目的に合う公開リポジトリ候補を比較し、短時間で判断できる検索体験を提供すること** である。

Gemini は次を常に意識すること。

- 一覧画面は比較画面である
- 詳細画面は最終判断支援画面である
- URL が検索状態の正本である
- 深いページ送りより検索条件の改善を優先する
- GitHub API の制約を前提に UI とロジックを組む
- API はサーバー側に閉じ込める
- Server Components を標準とし、Client は最小限に留める

---

## 4. 実装ルールの要約

以下は `AGENTS.md` セクション8 の要約である。詳細は正本を参照すること。

### やること
- 既存コードの命名規則に合わせる
- 既存コンポーネントや関数の再利用を優先する
- 純粋関数として切り出せる部分は切り出す
- UI から GitHub API を直接叩かない
- API の生フィールド名を UI にばら撒かない

### やらないこと
- 使い道が曖昧な抽象を作る
- 大規模リネーム
- 目的外の見た目変更を混ぜる
- テストが壊れているのに黙って終わらせる
