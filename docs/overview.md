# git-repo-finder 概要

## 全体サマリ

git-repo-finder は、GitHub の公開リポジトリを検索するアプリではなく、**目的に合う候補を比較し、短時間で判断できる検索体験を提供するアプリ**である。

---

## 1. 概要

git-repo-finder は、GitHub の公開リポジトリを検索し、比較し、目的に合う候補を判断しやすくするための Web アプリケーションである。
Next.js 14 以降 / App Router を前提に実装し、検索条件は URL クエリに保持する。入力途中の値はローカル state で扱い、検索として確定した状態のみを URL に反映することで、共有性・再訪性・ブラウザバック耐性・サーバー描画との整合性を確保する。

本アプリは「大量の結果を全件ブラウズする」ことではなく、**上位の一致候補から、使うべきリポジトリを素早く見つけること**を主目的とする。
そのため、UI も全件閲覧前提ではなく、比較と判断に最適化した構成を採用する。

---

## 2. 目的

本アプリの目的は、GitHub リポジトリ検索をシンプルに提供することではなく、**ユーザーが目的に合う候補を短時間で見つけ、比較し、判断できること**である。

これを実現するために、以下を両立する。

- URL 駆動の検索体験
- 一覧時点で比較しやすい情報設計
- GitHub API 制約に沿った現実的なページネーション
- Server Component を活かした安全な API 連携
- アクセシブルで迷いにくい UI
- テストしやすく保守しやすい構成
- Vercel 運用を前提にした軽量な観測設計　※なお、vercel限定というわけではない

---

## 3. プロダクト方針

### 3.1 プロダクトのコア価値

本アプリのコア価値は「検索できること」ではない。
**候補を見つける → 候補を比較する → 使うべき 1 件を判断する** という一連の流れを、GitHub 本体より軽く、迷いにくく支援することにある。

### 3.2 UX の基本方針

- 一覧画面は「検索結果表示」ではなく「比較画面」として設計する
- 詳細画面は「情報の再掲」ではなく「最終判断支援画面」として設計する
- 検索 API を目録ブラウザのようには扱わない
- 深いページ送りよりも、検索条件の見直しと上位候補の比較を重視する
- 数値指標だけでなく、説明文や更新状況を含めた文脈で判断できるようにする

### 3.3 捨てる前提

以下は本アプリの主目的に含めない。

- 全件を網羅的に閲覧すること
- 検索結果の最終ページまで移動すること
- GitHub 本体と同等以上の情報量をすべて再現すること
- 独自ランキングや推薦ロジックを持つこと

## 4. スコープ

### 4.1 実装対象

- 一覧画面
- 詳細画面
- GitHub API による公開リポジトリ検索
- URL クエリによる検索状態保持
- ページネーション
- 表示件数切替
- ローディング / 空状態 / エラー状態 / 404
- 基本的なログと指標計測
- SEO の index / noindex 制御
- 最近見たリポジトリ保存
- お気に入り保存
- README 内容表示

---

## 5. 想定ユーザーと主要ユースケース

### 5.1 想定ユーザー

- ライブラリやフレームワークの候補を探したい開発者
- 実装例や参考リポジトリを探したい開発者
- GitHub 検索は使うが、候補比較をもっと素早く行いたいユーザー

### 5.2 主要ユースケース

1. キーワードでリポジトリを検索する
2. 一覧で候補を比較する
3. 気になった候補の詳細を確認する
4. GitHub 本体へ遷移して中身を確認する
5. URL を共有して同じ検索状態を再現する
6. 気に入ったリポジトリをお気に入りに保存する
7. 最近見たリポジトリを振り返る
8. 詳細画面で README を確認する
