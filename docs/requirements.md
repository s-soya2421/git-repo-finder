# git-repo-finder 要件定義

## 1. 画面要件

### 1.1 一覧画面 `/`

#### 役割

一覧画面は、検索結果を単に返す画面ではなく、**候補を比較し、次に見るべきリポジトリを絞り込む画面**とする。

#### 表示要素

- タイトル: `git-repo-finder`
- 検索フォーム
- 検索条件の文脈表示
- 検索結果一覧
- 表示件数切替
- ページネーション
- 空状態 / エラー状態

#### 検索フォーム要素

- visible label 付き検索 input
- 検索ボタン
- 条件クリア導線
- 補助文
  - 例: `nextjs auth`
  - 例: `nextjs stars:>500 language:typescript`

#### 検索結果上部に表示する情報

- 検索見出し（例: `"nextjs auth" の検索結果`）
- 補助文（例: `上位の一致候補を表示中`）
- 表示範囲（例: `1–30件を表示`）
- 表示件数セレクト

#### 一覧項目

各リポジトリについて最低限以下を表示する。

- オーナーアイコン
- リポジトリ名
- オーナー名
- 説明文（先頭 50 文字を表示し、展開/折りたたみ可能）
- Topics（最大 3 件表示、超過分は `+N` で省略）
- License
- 主要言語
- Star 数
- 最終更新日
- コード最終更新日（`pushed_at`）
- Archived / Disabled 状態
- GitHub で開く導線
- 詳細へ進む導線

#### 一覧項目の情報優先順位

| 優先度 | 項目 |
|---|---|
| 第一 | リポジトリ名、説明文 |
| 第二 | Star 数、主要言語、最終更新日、コード最終更新日、Topics、License、Archived/Disabled |
| 第三 | オーナー、補助導線 |

#### 動作要件

- 説明文はデフォルト先頭 50 文字を表示し、クリックで全文展開/折りたたみができる
- Enter または検索ボタンで検索を実行する
- 自動検索は行わない
- 空文字では検索しない
- 検索条件は URL に保持する
- タイトルクリックでトップ初期状態へ戻る
- ブラウザバック時は検索状態を保持する
- 表示件数変更時は `page=1` に戻す
- 新しい検索実行時は `page=1` に戻す

#### 設計意図

一覧だけである程度の比較が完結することを重視する。
ユーザーが毎回詳細画面に遷移しなくても、「これは見に行く価値がありそう」と判断できる密度を持たせる。

---

### 1.2 詳細画面 `/repositories/[owner]/[repo]`

#### 役割

詳細画面は、一覧で気になった候補について、**GitHub 本体へ遷移する前の最終判断を支援する画面**とする。

#### 表示要素

- タイトル: `git-repo-finder`
- オーナーアイコン
- リポジトリ名
- オーナー名
- 説明文（全文表示）
- 主要言語
- Star 数
- Watcher 数
- Fork 数
- Issue 数
- License
- Topics（全件表示）
- 最終更新日
- コード最終更新日（`pushed_at`）
- Archived / Disabled 状態
- SECURITY.md 有無
- Dependabot 設定有無
- CI 状態（直近 workflow）
- Homepage
- README（Markdown レンダリング）
- GitHub で開く導線
- 一覧へ戻る導線

#### 動作

- 現状は独立ページとして表示（Intercepting Routes によるオーバーレイ表示は将来検討）
- タイトルクリックで一覧トップに戻る
- 戻る操作で検索結果文脈を失わないことを重視する
- ブラウザバックで一覧に戻る

#### 設計意図

詳細画面は数値の再掲だけにしない。
「何の repo か」「今も保守されているか」「次にどこを見るべきか」が分かる構成とする。

## 2. URL 設計

検索状態の正本は URL とする。
App Router では `searchParams` によりサーバー側で検索条件を取得でき、共有・復元・分析との相性が良い。

### 採用するパラメータ

| パラメータ | 役割 |
|---|---|
| `q` | 検索キーワード |
| `page` | ページ番号 |
| `perPage` | 1ページあたりの表示件数 |

例: `/?q=nextjs&page=2&perPage=30`

### ルール

- 入力途中の値はフォーム内部のローカル state で保持する
- 検索確定時のみ URL を更新する
- 検索実行時は `page=1` に戻す
- `perPage` 変更時も `page=1` に戻す
- `q` が空のときは検索を実行しない
- `page=1` やデフォルト `perPage` は URL から省略してよい

### バリデーション / 正規化

- `q`: trim 後 1 文字以上
- `page`: 1 未満は 1 に丸める
- `perPage`: 許可値以外はデフォルトへ丸める
- 不正パラメータは 400 ではなく正規化を優先する
- 必要に応じて正規化後 URL へリダイレクトする

## 3. 検索・ページネーション要件

### 3.1 基本方針

本アプリのページネーションは、全件を順番に閲覧するためのものではなく、**上位候補から比較対象を探すためのもの**とする。

### 3.2 GitHub Search API の制約

`search/repositories` は以下の制約がある。実装前に必ず考慮すること。

- **取得上限: 1000件**（`per_page × page ≤ 1000`）
  - 例: perPage=50 → 最大20ページ、perPage=10 → 最大100ページ
- 末尾ページリンクや総ページ数を前面に出さない理由のひとつはこの制約による
- 上限を超えるページへのアクセスは API エラーになるため、UI 側でページ上限を計算してガードする

### 3.3 ページネーション UI

採用形式: `< 2 3 [4] 5 6 >`

- 前へ / 次へ
- 近傍ページ群

採用しないもの:
- 末尾ページリンク
- 巨大な総ページ数の強調表示
- 全件閲覧を期待させる UI

### 3.4 表示件数

- 10 / 30 / 50
- GitHub REST API の `per_page` は最大 100、デフォルト 30
- UX と情報密度のバランスを優先し、MVP では上記 3 値に絞る

### 3.5 件数表示の考え方

- 総件数は参考情報として扱う
- 総ページ数を強く前面に出さない
- 「上位結果を表示中」といった文脈を添える
- `incomplete_results=true` の場合は、結果が一部のみの可能性を補足する

### 3.6 検索支援

検索結果が多い場合は、ページを深く進ませるよりも、検索条件の見直しを促す。

例:
- キーワードを短くする
- 別の語に変える
- `language:` や `stars:` などの qualifier を見直す

## 4. データ取得要件

### 4.1 API

| 用途 | エンドポイント |
|---|---|
| 一覧 | `GET /search/repositories` |
| 詳細 | `GET /repos/{owner}/{repo}` |
| README | `GET /repos/{owner}/{repo}/readme` |
| セキュリティポリシー有無 | `GET /repos/{owner}/{repo}/contents/{path}` |
| Dependabot 設定有無 | `GET /repos/{owner}/{repo}/contents/.github/dependabot.yml` |
| CI 状態 | `GET /repos/{owner}/{repo}/actions/runs` |

`Accept: application/vnd.github+json` を使用し、`X-GitHub-Api-Version` は定数で固定する。

### 4.2 実装方針

- GitHub API 呼び出しは `shared/github` に集約する
- API レスポンスから ViewModel への変換も同層に集約する
- `page.tsx` は `searchParams` の受け取り、取得、描画接続に専念する
- UI から GitHub API を直接叩かない
- クライアントへトークンを露出させない

### 4.3 責務分離

| 層 | 責務 |
|---|---|
| Page | URL / `searchParams` の受け取り、サーバー側取得 |
| Feature | UI 構築と画面ロジック |
| Lib | API 呼び出し、整形、バリデーション、純粋関数 |

## 5. Watcher 数の定義

画面ラベルは **Watcher** とする。
内部実装では GitHub REST API の `subscribers_count` を Watcher 数として扱う。

| 層 | 名称 |
|---|---|
| API DTO | `subscribers_count` |
| ViewModel | `watchers` |
| UI Label | `Watcher` |

---

## 6. ローディング・状態遷移要件

### 方針

- 一覧画面・詳細画面ともにスケルトン UI を出し分ける
- スピナー単体は基本採用しない

### 理由

- レイアウトの見通しがつきやすい
- 体感性能が上がる
- レイアウトシフトを抑えやすい
- 一覧比較の画面構造を維持しやすい

---

## 7. エラー・空状態要件

### 7.1 分類

- 検索結果 0 件
- GitHub API エラー
- primary rate limit
- secondary rate limit
- 一時的な upstream failure
- 不正な URL パラメータ
- リポジトリ未存在時の 404

### 7.2 表示方針

| ケース | メッセージ |
|---|---|
| 0 件 | `該当するリポジトリが見つかりませんでした` |
| 0 件補助文 | `キーワードを短くする、別の語に変える、条件を減らすと見つかる場合があります` |
| レート制限 | `アクセスが集中しています。少し時間をおいて再度お試しください` |
| API 失敗 | `データの取得に失敗しました` |
| 404 | `リポジトリが見つかりませんでした` |

### 7.3 実装方針

- ユーザー向けメッセージと運用向けログを分離する
- 404 と API 失敗を混同しない
- rate limit では `retry-after` があればそれを考慮する
- primary / secondary rate limit をログで識別できるようにする

## 8. アクセシビリティ要件

UI ライブラリは shadcn/ui を採用しつつ、アプリ全体の意味づけ・操作性・可読性を要件として担保する。

### 最低ライン

- 検索 input に visible label を付与
- キーボードのみで全操作可能
- ページネーションに `aria-current="page"` を付与
- 画像の `alt` を明示
- フォーカスリングを消さない
- 色だけで意味を伝えない
- ボタン / リンクの役割を曖昧にしない

### 補足

- 検索フォームは submit 可能な `form` とする
- タップ領域は十分なサイズを確保する
- 一覧カード全体をクリック可能にする場合は、内部リンクと競合しないようにする
- スケルトンは視覚的補助であり、必要に応じて状態説明文も併用する
- 補助情報はアイコンだけに頼らない

## 9. SEO 要件

### 9.1 index / noindex 方針

| ページ | 方針 |
|---|---|
| `/` トップページ | index |
| `/?q=...` 検索結果 | noindex |
| `/repositories/[owner]/[repo]` | 当面 noindex |

### 9.2 理由

- 検索結果 URL が無制限に増えやすい
- 外部データの薄い再掲ページを量産しない
- 低品質ページ評価や SEO ノイズを避ける

### 9.3 実装方針

- トップは静的 metadata で `robots.index = true`
- 一覧は `q` の有無で `generateMetadata` により制御する
- 詳細は当面 noindex
- 404 は `notFound()` を用い、専用 not-found UI を表示する

## 10. お気に入り要件

### 10.1 概要

ユーザーが気に入ったリポジトリを手元に保存し、後から見返せるようにする。

### 10.2 保存先

- **LocalStorage**（ログイン不要、ブラウザ間同期なし）
- キー: `git-repo-finder:favorites`

### 10.3 保存上限

- 上限なし（LocalStorage の容量制限に依存）

### 10.4 保存データ

リポジトリの再取得なしに一覧表示できる最小限の情報を保持する。

| フィールド | 内容 |
|---|---|
| `id` | リポジトリ ID |
| `owner` | オーナー名 |
| `repo` | リポジトリ名 |
| `description` | 説明文 |
| `language` | 主要言語 |
| `stars` | Star 数（保存時点） |
| `savedAt` | 保存日時 |

### 10.5 表示場所

- 詳細画面: お気に入り追加/解除ボタン
- 専用ページ `/favorites`: 保存済み一覧

### 10.6 動作

- 追加/解除はトグル形式
- お気に入り済みかどうかは詳細画面で視覚的に区別する
- `/favorites` で個別削除・全削除が可能

---

## 11. 最近見たリポジトリ要件

### 11.1 概要

ユーザーが詳細画面を開いたリポジトリを自動記録し、後から振り返れるようにする。

### 11.2 保存先

- **LocalStorage**（ログイン不要、ブラウザ間同期なし）
- キー: `git-repo-finder:recently-viewed`

### 11.3 保存上限

- 最大 **20件**（超過時は古いものから削除）

### 11.4 保存データ

お気に入りと同じ最小限の情報を保持する（`savedAt` → `viewedAt`）。

### 11.5 表示場所

- トップページ（検索前の初期状態）: 最近見た一覧をコンパクトに表示
- 専用ページは設けない

### 11.6 動作

- 詳細画面を開いた時点で自動記録
- 同じリポジトリを再訪した場合は `viewedAt` を更新して先頭に移動
- 個別削除・全削除が可能
